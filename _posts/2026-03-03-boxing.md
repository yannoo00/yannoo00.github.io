### Boxing  

함수가 끝날 때 함께 소멸하는 메모리를 저장하는 stack에는 value type이 올라가고,  
runtime 중 동적 할당 되는 heap에는 reference type이 올라간다.  

ref type에는 값을 가지고 있는 데이터의 '주소'가 저장된다.  
어떤 함수에서
```cs
{
  ... 
  ArrayList arryList = new ArrayList();
}
```  
처럼 arrayList를 동적할당 했다고 하자. 변수 arrayList에는 new ArrayList로 생성한 ref type ArrayList의 주소가 저장된다.  
그 주소를 따라가면 heap에 실제 데이터가 저장되어 있을 것이다.  

반면 
```cs  
int i = 10;
```
처럼 작성하면 stack에 10이라는 값을 올리게 된다.  
이 int는 value type이다.  

이 때 stack에 저장된 데이터를 복사하여 힙에 올리는 것을 boxing이라고 하고, heap의 데이터가 stack에 복사되어 저장되는 것을 unboxing이라고 한다.   
왜 이런 현상이 발생할까?  

C#에서 대부분의 데이터 타입은 object를 상속받는다. object는 C# 객체가 지원하는 기본적인 기능을 갖춘 객체이자 ref type이다.  
문제는 value type인 int, struct와 같은 자료형도 object를 상속 받는다는 것이다.  

```cs
int i = 123;
// The following line boxes i.
object o = i;

//또는 명시적으로..
int i = 123;
object o = (object)i;  // explicit boxing
```  
i는 value type으로 stack에 저장되었지만 o에 할당하려는 순간 boxing을 거쳐 heap에 올라가게 된다.  
o는  heap에 저장된 123이라는 값을 가리키는 주소값이 된다.  


```cs
// List example.
// Create a list of objects to hold a heterogeneous collection
// of elements.
List<object> mixedList = [];

// Add a string element to the list.
mixedList.Add("First Group:");

// Add some integers to the list.
for (int j = 1; j < 5; j++)
{
    // Rest the mouse pointer over j to verify that you are adding
    // an int to a list of objects. Each element j is boxed when
    // you add j to mixedList.
    mixedList.Add(j);
}
```  

위처럼 heap에 있는 mixedList에 value type인(그래서 stack에 있는) 객체를 저장하려면 boxing을 거쳐야 한다.  


```IL
// j를 스택에 로드 (예: j가 1일 때)
ldloc.s    j          // 로컬 변수 j의 값을 스택에 쌓음 (Value Type)

// 박싱 수행
box        [System.Runtime]System.Int32 // 스택 위의 int를 객체로 감싸고 주소(참조)를 반환

// List<object>.Add 메서드 호출
callvirt   instance void class [System.Collections]System.Collections.Generic.List`1<object>::Add(!0)
```  
IL을 보면 위와 같다. 스택에 올린 j에 boxing을 수행하여 힙에 메모리를 할당하고 값을 복사한 뒤 그 주소를 Add의 인자로 전달한다.  
이처럼 IL에서는 box 키워드를 사용해서 명시적으로 박싱/언박싱이 일어난다. 메모리 할당, 값 복사 연산을 해야 하기에 성능에 부정적일 수 있다.
