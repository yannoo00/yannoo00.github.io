### Evnet Bus 패턴  

이벤트의 주체가 이벤트를 발행시키고, 중간의 이벤트 버스를 통해 이벤트 발생에 따른 행위의 주체들이 이벤트를 구독하는 방식으로 편리하게 && 결합도를 낮추는 방식으로 이벤트에 따른 변화를 구현할 수 있다.  

이벤트 버스 패턴은 크게 Publisher / Event Bus / Subscriber 세 가지 주체로 구성된다.  

Publisher는 어떤 일이 발생했을 때 - 즉 내가 '이벤트'로 볼 어떤 사건이 발생했을 때 그 사건이 '발생했음'을 이벤트 버스에게 알린다. 정말 publish만 하고, 더 이상은 신경쓰지 않는다.  

Event Bus는 Publisher가 발행한 이벤트를 구독자들에게 전달한다.  

Subscriber들은 publishser가 발행할 어떤 일 - 즉 이벤트들을 미리 구독하고 있는다. 이벤트 버스는 이렇게 각 이벤트를 구독중인 구독자들에게 이벤트 발생 사실을 전달한다. 그러면 subscirber들은 이벤트 발생에 따른 행동 지침에 따라 정해진 동작을 수행한다.  

이처럼 구현하게 되면 이벤트를 발행시키는 쪽과 그에 따른 행위를 하는 쪽이 서로를 모르기 때문에 유연한 확장과 이벤트 1:N 연결에도 용이하다.  


어떻게 구현할 수 있을지 간단하게 보자.  
```javascript
//이벤트 버스
const EventBus = {
  listeners: {},

  // 구독
  on(event, callback) {
    if (!this.listeners[event]) this.listeners[event] = [];
    this.listeners[event].push(callback);
  },

  // 발행
  emit(event, data) {
    if (this.listeners[event]) {
      this.listeners[event].forEach(callback => callback(data));
    }
  }
};

// 2. 구독자 (Subscriber) 
EventBus.on('userLoggedIn', (username) => {
  console.log(`[알림] ${username}님이 로그인하셨습니다.`);
});

// 3. 발행자 (Publisher) - "로그인 처리" 컴포넌트라고 가정
function loginProcess() {
  // ...로그인 로직 성공 후...
  EventBus.emit('userLoggedIn', '성현우');
}

// 실행
loginProcess(); // 출력: [알림] 성현우님이 로그인하셨습니다.
```  

구독하는 쪽에서 callback으로 'userLoggedIn'이 발생했을 때 실행할 행위를 지정해줬다.  
이벤트 Publisher에서는 로그인 프로세스 처리 이후 emit()을 통해 등록된 모든 콜백을 실행한다(실행할 콜백이 많은 경우 테스크 큐에 넣거나  하는 방식을 사용하는게 나을 것 같다 아무튼 예시니까 패스).  

그럼 이 패턴의 단점은 뭘까?  
일단 위의 방식과 비슷하게 직접 코드를 작성하다보면 가장 불편한 점이, 위 코드로 치면 'userLoggedIn'이라는게 함수가 아니기에 코드 흐름을 보고 싶어도 F12로 함수 정의 이동이 안되는 것이다...  
그 외에도 이 로그인 로직 관련해서 에러가 발생했을 때 - 디버깅이 더 어려워진다. 하나의 함수 로직 안에서 발생하는게 아니니, 여러가지 구독자들의 코드를 다 보면서 어디가 원인일지 찾아야 할 수 있다.  

그래도 ButtonClick에 따라 서너가지 변화가 UI든 어디에서든 있어야 하는 상황이라고 할 때 이런 패턴 없이 정말 하나 하나 함수로 A가 일어났을 때 B를 하고... 이런 식으로 구현하게 되면 코드도 너무 많아지고 작은 수정에도 대응하기 힘들어진다.  
또한 이벤트 자체를 하나의 명확한 클래스로 정의하고, 구독자가 인터페이스를 받아 이벤트 발생에 따른 handler를 구현하는 방식으로 코드를 작성하는 등의 응용이 가능하다.  

```C#
using System;
using System.Collections.Generic;

namespace SmartEventBus
{
    // 1. 이벤트 데이터를 담을 명확한 '메시지 클래스' 정의
    public class OrderPlacedMessage 
    {
        public string ProductName { get; set; }
        public int Price { get; set; }
    }

    // 2. 이벤트를 처리할 구독자 전용 '인터페이스'
    public interface IHandle<TMessage>
    {
        void Handle(TMessage message);
    }

    // 3. 개선된 이벤트 버스
    public class MessageBus
    {
        private readonly List<object> _subscribers = new();

        // 객체 자체를 구독자로 등록
        public void Subscribe(object subscriber) => _subscribers.Add(subscriber);

        // 메시지 타입에 맞는 구독자를 찾아 실행
        public void Publish<TMessage>(TMessage message)
        {
            foreach (var subscriber in _subscribers)
            {
                // 구독자가 해당 메시지를 처리할 수 있는 인터페이스(IHandle)를 구현했는지 확인
                if (subscriber is IHandle<TMessage> handler)
                {
                    handler.Handle(message);
                }
            }
        }
    }

    // 4. 구독자 구현 (인터페이스 상속)
    public class EmailService : IHandle<OrderPlacedMessage>
    {
        public void Handle(OrderPlacedMessage message)
        {
            Console.WriteLine($"[이메일] {message.ProductName}({message.Price}원) 영수증 발송");
        }
    }

    class Program
    {
        static void Main()
        {
            var bus = new MessageBus();
            var emailService = new EmailService();

            // 구독 등록
            bus.Subscribe(emailService);

            // 발행 (문자열이 아닌 '객체'를 통째로 던짐)
            bus.Publish(new OrderPlacedMessage { ProductName = "아이패드", Price = 1200000 });
        }
    }
}
```
