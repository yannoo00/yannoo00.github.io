IOCP가 성능이 좋은(=빠른) 이유 중 한 가지로 DMA를 뽑을 수 있다.  

먼저, DMA가 없을 때 프로세스에서 네트워크를 타고 들어온 데이터를 받는 과정을 생각해보자.  

NIC에 패킷이 들어온다. NIC 내부에도 데이터를 담아야하니 버퍼가 있을 것이다. 여기에 패킷이 쌓인다.  
데이터 처리를 위해 NIC이 인터럽트를 발생시킨다.  

인터럽트가 발생했으니, CPU는 Context Switch를 실행하여 NIC의 데이터를 가져오는 일을 수행한다.  
NIC의 버퍼에 있던 데이터를 CPU 레지스터로 가져온 다음 커널 영역의 버퍼에 쓰는 작업을 통해 - HW 계층에서 커널 영역으로 데이터를 올린다.  

데이터가 커널 메모리로 다 올라왔다면, OS가 패킷 헤더를 뜯어보고(IP -> TCP) 이 패킷이 어떤 포트와 소켓을 향하는지 식별하여  해당 소켓의 큐에 패킷을 전달한다.  

유저 프로세스는 데이터를 읽기 위해 소켓에 recv(socket, ... ) 같은 시스템 콜을 호출한다. 시스템 콜이 발생하면 유저 모드에서 커널 모드로 컨텍스트 스위칭이 발생하고 -> 커널 버퍼에 데이터가 준비된 경우 CPU가 다시 메모리 복사 연산을 통해 소켓 버퍼의 데이터가 유저 프로세스가 '여기 데이터를 쓰라'고 넘긴 메모리 주소에 복사된다.  
그럼 이제 다시 시스템 콜이 리턴 되면서 유저 모드로 돌아와 들어온 패킷을 파싱하고 데이터에 따른 처리를 진행한다.  

즉 메모리를 HW/Kernel/User 계층을 오가면서 메모리에 몇 번이고 CPU를 이용하여 복사하는 과정이 필요하다.  
DMA를 활용하는 IOCP는 하드웨어가 이미 써야하는 최종 메모리 주소를 알게 된다.  

유저 스레드에서 WSARecv(socket, user_buffer, ... )를 호출하면, 앞으로 이 socket으로 오는 데이터는 user_buffer에 바로 쓰라는 요청이다.  
I/O 매니저는 가상 메모리에 있는 user_buffer의 실제 물리 주소를 찾아내고 이 유저 버퍼 메모리의 주소가 바뀌지 않도록 RAM에 고정(Pin)시킨다. 네트워크 드라이버는 유저 버퍼의 물리 주소를 DMA 컨트롤러의 레지스트리에 세팅해둔다.  

다시금 NIC에 패킷이 들어왔을 때 - 이번에는 CPU 개입 없이 DMA컨트롤러가 NIC의 버퍼에 있는 데이터를 RAM의 user_buffer로 바로 써준다. 즉 CPU 개입 한 번 없이 프로세스의 buffer에 데이터가 써졌다. 이후 DMA 컨트롤러가 데이터를 다 썼다는 인터럽트를 발생시키면 I/O Manager가 데이터를 체크하고  유저 버퍼에 잡혀있던 고정을 해제하고, IOCP 완료 큐에 패킷이 들어간다.  

기본적으로 패킷이 들어올 때까지 메모리 고정 -> 데이터 수신 완료 후 해제 -> 다시 다음 패킷을 기다리며 대기 및 버퍼 메모리 고정  ... 을 반복하지만 Registered I/O는 대용량 유저 버퍼를 커널에 영구적으로 1번 등록하여 패킷을 계손 쏘게 된다.
