Classful IPv4 주소체계에서는 값의 맨 앞 8비트만 보고 클래스를 식별, 어디까지가 네트워크 주소이고 어디까지가 호스트 주소인지를 알 수 있다.  

A클래스는 1.0.0.0~126.255.255.255의 범위(0.0.0.0과 127. 대역은 각각 브로드캐스트/루프백 주소 값. 그래서 웹서버 열고 허용 IP 주소 설정할 때 0.0.0.0으로 설정하면 같은 네트워크에서 오는 모든 패킷을 받을 수 있게 된다. 기본은 아마 루프백 주소만 허용되어 있을 것이다.)  
B클래스는 128.0.0.0~191.255.255.255의 범위  
C클래스는 192.0.0.0~223..  
C~A로 가면서 호스트 수가 많아지는 구조다.  

하지만 이 구조는 주소가 낭비되는 문제가 있어서 서브넷 마스크를 도입하고, 라우팅 할 때 클래스를 사용하지 않고 서브네팅을 하여 호스트를 쪼개게 되었다.  
예를 들어 어떤 회사에서 400대의 컴퓨터가 필요하다면 - C 클래스 하나를 할당 받기에는 가능한 호스트 수가 254개로 부족하다.  
그렇다고 B클래스는 호스트가 65,534개로 B클래스를 쓰기에는 낭비가 심하다.  
그럼 C클래스 두개를 받으면? 이것도 결국 사용되는 ip 주소의 개수는 줄겠으나 결국 '다른' 네트워크가 되기 때문에 사내 네트워크가 분리되고 회사에서 Router를 거쳐야하는 비효율이 발생한다.  

CIDR(Classless Inter-Domain Routing)은 A,B,C처럼 정해진 호스트 개수를 사용하지 않고, IPv4 주소 뒷자리에 192.168.0.15/24 처럼 서브넷 마스크 값을 붙인 것으로 판단하는 라우팅 방식이다.  
다시 위의 상황에서 400대를 쓰기 위해 이번에는 서브네팅을 활용한다고 하자. 그럼 C클래스의 경우 원래 하나의 네트워크에서 2^8 = 256개의 네트워크가 사용 가능했다면, 400에 근접한 2^9 = 512개를 사용하면 된다. 이 경우 서브넷 마스크는 255.255.254.0이 된다.  
이진수로 보면 11111111.11111111.11111110.0으로 3번째 8비트? 의 첫번째 값을 하나 더 호스트 주소로 할당해주면서 0이 된 것을 알 수 있다.  
이러면 이제 낭비가 획기적으로 줄어들고 라우팅 효율도 상승한다.
