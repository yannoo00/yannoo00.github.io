오라클의 아키텍쳐는 DB/Memory/process로 분류할 수 있다.  

<img width="1224" height="688" alt="Image" src="https://github.com/user-attachments/assets/d84784e6-bf2f-4123-8b09-4729bd2d4778" />  

DB 영역은 실제 정보가 파일이라는 실체로 저장되는 공간이다. file 종류로는 data file, redo file, control file, ... 등이 있다.  
테이블 스페이스 > 세그먼트 > 익스텐트 > 블록 순서의 계층 구조로 저장된다.  

- 테이블스페이스: 논리적 저장 구조 중 최상위 개념으로, 물리적 디스크 파일인 데이터 파일과 1:N 관계로 매핑된다. 하나의 테이블스페이스는 여러 데이터 파일로 구성될 수 있으나, 하나의 데이터 파일은 단 하나의 테이블스페이스에만 속한다. 테이블스페이스를 구성하는 데이터 파일들은 디스크 상에서 물리적으로 분리되거나 불연속적으로 존재할 수 있다.  
- 세그먼트: 테이블, 인덱스, Undo 정보(구버전 테이블 정보?) 등 고유한 물리적 저장 공간이 필요한 데이터베이스 객체(ex: '사원' 테이블, 'emp' index, ... )당 하나씩 생성되는 단위이다. 하위 계층 익스텐트의 집합으로, 익스텐트들은 불연속적으로 존재한다.  
- 익스텐트: 오라클이 저장공간을 확장할 때 사용하는 최소 단위. 하나의 익스텐트를 구성하는 블록들은 물리적으로 연속되어 저장된다. 디스크 헤더의 이동을 최소화하여 멀티 블록 I/O 성능을 높인다.  
- 데이터 블록: I/O 및 메모리 전송의 최소 단위. 디스크에서 SGA의 데이터베이스 버퍼 캐시로 데이터를 읽거나 캐시에서 디스크로 dirty block 등을 쓸 때 이 단위를 쓴다.  


오라클 인스턴스가 가동될 때 OS에게 할당받는 공유 메모리 영역을 SGA(system global area)라고 한다.  
여기에는 캐시/shared pool 등이 있다.  
- DB 버퍼 캐시: 디스크에서 읽은 실제 데이터 캐싱
- Shared pool: SQL 실행 계획을 담는 라이브러리 캐시, 메타데이터를 담는 딕셔너리 캐시 등 
- redo log 버퍼: 변경 이력을 redo log file로 내리기 전 임시 저장소
